# Core Claim

A data science product should connect four things:

1. **Question**
2. **Data**
3. **Code**
4. **Claim**

<br>

Reproducible work keeps that chain visible.

If the reader cannot trace the claim back to the code and data, the result is weaker than it looks.

---

# Why This Matters for Urban Data Science

Urban data products often inform public decisions.

- Which corridors need safety investments?
- Which neighborhoods face unequal exposure?
- Which transit stops should be prioritized?
- Which policy change appears to have worked?

<br>

The stakes make "I ran some code and got this chart" too thin. The work needs a trail.

---

# Reproducibility Is a Spectrum

| Level | Meaning | Class example |
| :--- | :--- | :--- |
| **Repeatable** | You can rerun it on your laptop | Your notebook runs top-to-bottom |
| **Reproducible** | Someone else can rerun it | README + environment + data instructions |
| **Robust** | Result survives reasonable choices | Same pattern under alternative filters |
| **Auditable** | Reader can inspect assumptions | Narrative explains decisions and limits |

<br>

For this class, aim for reproducible and auditable.

---

# What Counts as Reproducible Here?

Before submitting an assignment or project milestone:

- Raw data source is named and linked
- Cleaning steps are in code, not manual edits
- Notebook or Quarto document runs from top to bottom
- Figures and tables are generated by code in the document or project
- Package requirements are documented
- AI use is disclosed and checked

---

# Computational Narrative

A notebook or Quarto report is not just code with comments.

It should read like:

1. What question are we asking?
2. What data are we using?
3. What transformations did we make?
4. What did we find?
5. How confident should we be?
6. What should happen next?

<br>

Markdown cells carry the argument. Code cells carry the evidence.

---

# The Bad Version

```python
df = pd.read_csv("data.csv")
df = df[df.x > 0]
df.groupby("area").mean().plot()
```

<br>

What is missing?

- What is `x`?
- Why drop values below zero?
- What does `area` represent?
- Is the mean the right summary?
- What does the chart imply?

---

# The Better Version

```markdown
## Are injury crashes concentrated on high-speed corridors?

The ODOT crash file includes a posted speed limit field. I exclude
records with missing or zero speed limits because those values do not
represent an observed corridor speed.
```

```python
valid_speed = crashes["POST_SPEED_LMT_VAL"].between(5, 80)
speed_summary = (
    crashes.loc[valid_speed]
    .groupby("POST_SPEED_LMT_VAL")["TOT_INJ_CNT"]
    .mean()
)
```

# A Simple Report Shape

For most class projects:

| Section | Purpose |
| :--- | :--- |
| Question | State the urban problem clearly |
| Data | Source, coverage, unit of observation |
| Methods | Cleaning, filters, transformations |
| Findings | Figures/tables plus interpretation |
| Limitations | What the data cannot answer |
| Next steps | What analysis or action follows |

<br>

This structure works for notebooks, Quarto reports, and dashboards.

# Part 2: Jupyter Notebooks

---

# What Jupyter Is Good For

Jupyter notebooks are excellent for:

- Interactive data exploration
- Teaching and showing code incrementally
- Mixing text, code, output, and charts
- Trying multiple approaches quickly
- Sharing a compact analysis artifact

<br>

Notebooks are a workspace and a communication format. That dual role is useful, but it creates risks.

---

# Notebook Failure Modes

- Cells run out of order
- Hidden state from deleted or old cells
- Long notebooks with no section structure
- Large outputs committed into version control
- Important cleaning logic buried in one cell
- "Works after I click around" but not from a clean restart

<br>

Rule: if restart-and-run-all fails, the notebook is not done.

---

# Execution Order Matters

This notebook can produce a result that cannot be reproduced:

```python
# Cell 5
crashes = crashes[crashes["CNTY_NM"] == "MULTNOMAH"]
```

```python
# Cell 2, run later
crashes = crashes.sample(1000)
```

```python
# Cell 9
crashes.groupby("CRASH_SVRTY_LONG_DESC").size()
```

<br>

The visible order and the executed order are not always the same.

---

# Notebook Hygiene Checklist

Use this before submitting:

- Restart kernel
- Run all cells from top to bottom
- Remove dead experiments or label them clearly
- Put imports and settings at the top
- Use section headings
- Save important outputs to `outputs/`
- Keep reusable functions in `.py` files when they grow

---

# Suggested Notebook Skeleton

```markdown
# Crash Severity Exploration

## Setup
Imports, paths, display options

## Load Data
Source and file path

## Clean Data
Type fixes, filters, missing values

## Explore
Tables and plots

## Findings
Short written interpretation
```

---

# Naming Helps

Bad names make notebooks harder to audit.

```python
df2 = df[df["x"] > 0]
tmp = df2.groupby("y").size()
tmp.plot()
```

Better:

```python
valid_speed_crashes = crashes[
    crashes["POST_SPEED_LMT_VAL"].between(5, 80)
]

crashes_by_speed = valid_speed_crashes.groupby(
    "POST_SPEED_LMT_VAL"
).size()
```

---

# Move Stable Logic Out

When a cleaning cell gets long, move it into a function.

```python
# crash_cleaning.py
def clean_crashes(crashes):
    out = crashes.copy()
    out["CRASH_DT"] = pd.to_datetime(out["CRASH_DT"], errors="coerce")
    out = out.dropna(subset=["LAT", "LON"])
    return out
```

```python
from crash_cleaning import clean_crashes

crashes = clean_crashes(raw_crashes)
```

<br>

The notebook becomes easier to read; the function becomes easier to test.

---

# Exploration vs Submission

| Exploration notebook | Submission notebook/report |
| :--- | :--- |
| Messy is acceptable | Must run cleanly |
| Many rough plots | Fewer selected figures |
| Notes to yourself | Narrative for a reader |
| Tries alternatives | Explains chosen path |
| Temporary outputs | Stable outputs |

<br>

You do not need to preserve every dead end in the final document.

# Part 3: Quarto

# What Quarto Adds

Quarto turns code plus markdown into polished documents.

- Source file: `.qmd` or `.ipynb`
- Outputs: HTML, PDF, Word, RevealJS slides, websites
- Runs Python, R, Julia, Observable
- Supports figures, tables, citations, parameters, themes
- Works well with Jupyter kernels

<br>

For this class: Quarto is a strong default for final reports.

---

# Quarto File Anatomy

```yaml
title: "Crash Patterns Near Schools"
author: "Your Name"
format: html
jupyter: python3
```

```markdown
## Question

Which school areas have the highest concentration of severe crashes?
```

```python
import pandas as pd
import geopandas as gpd
```

<br>

YAML controls document settings. Markdown controls narrative. Code controls computation.

---

# Render Command

```bash
quarto render crash_report.qmd
```

Useful variants:

```bash
quarto render crash_report.qmd --to html
quarto render crash_report.qmd --to pdf
quarto preview crash_report.qmd
```

<br>

HTML is the easiest default. PDF may require a LaTeX installation.

---

# Code Cell Options

Put options at the top of a Quarto code cell:

```python
#| label: fig-hourly-crashes
#| fig-cap: "Crash count by hour of day, Oregon 2023"
#| echo: false

sns.countplot(crashes, x="CRASH_HR")
```

| Option | Meaning |
| :--- | :--- |
| `echo: false` | Hide code, show output |
| `warning: false` | Hide warnings |
| `message: false` | Hide package messages |
| `fig-cap` | Add a figure caption |
| `label` | Stable reference name |

---

# When to Show Code

Show code when:

- The audience needs to inspect methods
- The code is short and clarifies the result
- The assignment asks for code

Hide code when:

- The output is the focus
- The code is long and distracts from the finding
- You are producing a public-facing report

<br>

For this class, include enough code that I can verify your workflow.

```python
import matplotlib.pyplot as plt
```

<br>

Polish figures in code. Do not edit exported images manually.

# Tables in Quarto

```python
summary = (
    crashes

    .agg(
        crashes=("CRASH_ID", "count"),
        mean_injuries=("TOT_INJ_CNT", "mean"),
    )

summary
```

Small tables belong in reports. Large tables usually belong in downloadable files.

# Citations and Links

At minimum, link your data sources in the text.

```markdown
Crash records come from the
[Oregon Department of Transportation crash data products](https://www.oregon.gov/odot/Data/Pages/CrashDataProducts.aspx).
```

For formal papers, Quarto can use BibTeX:

```yaml
bibliography: references.bib
```

```markdown
Prior work links nighttime conditions to pedestrian risk [@nyt_pedestrians_2023].
```

---

# Parameters

Parameters let you rerun one report for different places or years.

```yaml
title: "Crash Summary"
params:
  county: "MULTNOMAH"
  year: 2023
```

```python
county_crashes = crashes[
    (crashes["CNTY_NM"] == params["county"]) &
    (crashes["CRASH_DT"].dt.year == params["year"])
]
```

<br>

This is useful for reusable city or corridor reports.

| :--- | :--- |

| Final report | Quarto |
| Public HTML report | Quarto |
| Interactive dashboard | Streamlit or Quarto dashboard |
| Static PDF handout | Quarto |

<br>

The tools overlap. Choose based on the audience and final artifact.

# Minimal Folium Export

```python
import folium

m = folium.Map(location=[45.52, -122.67], zoom_start=11)

for _, row in crashes.dropna(subset=["LAT", "LON"]).iterrows():
    folium.CircleMarker(
        location=[row["LAT"], row["LON"]],
        radius=3,
        popup=row["CRASH_SVRTY_LONG_DESC"],
        fill=True,
```
